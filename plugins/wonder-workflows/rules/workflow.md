# Development Workflow Meta-Rule

> Related meta-rules: `${CLAUDE_PLUGIN_ROOT}/rules/structure.md` · `${CLAUDE_PLUGIN_ROOT}/rules/security.md`

This document defines the **wonder-workflows 6-stage workflow**. It applies to every development task regardless of stack.

## The 6-Stage Pipeline (fixed order, no skipping)

| Stage | Agent | Input | Output |
|-------|-------|-------|--------|
| 1. Analysis | analyzer | task definition + `.claude/rules/` | `work-doc.md §Analysis` |
| 2. Research | researcher | §Analysis + template catalog | `work-doc.md §Research` (with injected templates + candidate markers) |
| 3. Planning | planner | §Analysis + §Research | `work-doc.md §Planning` |
| 4. Implementation | developer | §Planning + `.claude/rules/` | code files |
| 5. Inspection | inspector | code + §Planning + `.claude/rules/` | `inspection-report.md` |
| 6. Modification | modifier | `inspection-report.md` | `modification-report.md` + delegated fixes |

## Enforcement

This workflow is enforced by convention, not by hooks. The orchestrator drives stage ordering and each agent follows its documented stage permissions. Writing artifacts out of stage order should be avoided — the orchestrator is responsible for keeping stages in sequence in both command and prompt modes.

## Project Rules as the Stack Source

Each agent reads `.claude/rules/` to determine stack-specific behavior:
- `.claude/rules/{layer}.md` (one file per active layer present in `.claude/rules/`) — structural and layer conventions
- `.claude/rules/security.md` — security policies

These files are generated by `/wsf-init` from the project's actual codebase. They are the authoritative source — not the plugin's meta-rules.

## Deliverable Structure

```
.claude/
└── runs/{run-id}/
    ├── work-doc.md           ← §Analysis, §Research, §Planning
    ├── inspection-report.md  ← quality, correctness, security, rules findings
    └── modification-report.md  ← fix list and applied changes (Stage 6 only)
```

## Principles

- Every stage must produce its deliverable before the next stage begins.
- YAGNI: implement only what the task specifies.
- Project rules override general knowledge. If `.claude/rules/{layer}.md` says X, do X.
- Template catalog (managed by wonder-utilities) is read by researcher at Stage 2 if available. Promote candidates via `/wsu-template promote` after each run.
