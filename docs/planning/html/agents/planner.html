planner

요청 문서를 읽고 도메인을 7파일 모듈 계획으로 분해한다. 의존성과 구현 순서를 결정하는 파이프라인 첫 번째 단계.

개요

도메인 네이밍 공식을 적용하고 7파일 분해 계획을 생성한다. YAGNI 원칙 — 요청 범위를 벗어나는 기능은 추가하지 않는다.

백엔드 5파일

  • {Entity}Controller
  • {Entity}Service
  • {Entity}Mapper
  • {Entity}DTO
  • {Entity}Form

프론트엔드 2파일

  • {domainName}.js
  • {domainName}.html

현재 동작

  • 요청 문서(create 또는 modify)를 읽는다.
  • 도메인 네이밍 공식 적용: 모듈코드(2자) + camelCase 기능명 = woWorkShift.
  • 파일 간 의존성 파악 (예: Controller는 Service에 의존).
  • 순환 참조 없는 구현 순서 결정.
  • modify 모드: 변경 대상 파일만 포함.
  • review 모드: 계획 없이 기존 파일 검토만.

입력 / 출력

내용
입력.claude/requests/{create|modify}_request.md
출력모듈 계획 문서 (파일 목록·네이밍·의존성·구현 순서)

의존성

  • 요청 문서에 모든 필수 섹션이 있어야 한다.
  • modify 모드에서는 기존 코드를 읽어 네이밍 충돌을 감지한다.
  • 이 단계에서는 템플릿·규칙 의존성 없음.

개선 방향

유사한 기존 모듈을 감지하고 신규 생성 전 재사용을 권장.
다운스트림 에이전트가 안정적으로 파싱할 수 있도록 계획을 구조화 JSON으로 출력.
기존 모듈 코드와의 충돌 여부를 도메인 네이밍 검증 단계에서 확인.
modify 모드에서 "변경 없는 파일" 목록을 명시해 하위 에이전트가 건너뛰도록.
