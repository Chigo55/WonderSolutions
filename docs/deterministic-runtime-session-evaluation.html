종합 판정

CLI/MCP 공개 표면, 단일 operation registry, 16개 operation 등록, 테스트 통과, 0.2.0 버전 범프, hook 활성화는 유지했다. 추가로 스펙 파일 추적성, run scaffold 보존성, 런타임 결정성, typed update 엄격성, reuse output 쓰기 operation을 보완했다.

Findings And Remediation

Resolved 핵심 스펙 파일 추적성

초기 평가에서는 docs/deterministic-runtime.md가 미추적이라 release completeness가 깨져 있었다. 이번 보완 커밋에 스펙 문서를 포함해 코드 주석과 테스트가 참조하는 계약 문서가 원격에도 남도록 한다.

Remediation: docs/deterministic-runtime.md를 커밋 대상에 포함.

Resolved createRunScaffold 기존 run 파일 보존

스펙은 run record를 “create once”로 정의하고 기존 user/agent content 보존을 기본값으로 둔다. 이번 보완에서 공통 writeRunFilesIfAbsent helper를 추가해 build/govern/reuse/extend scaffold가 기존 파일을 덮지 않고 existingPaths로 보고하게 했다.

Remediation: tools/shared/runtime/run-files.ts 추가, operations.ts path disposition 수정, registry-level preservation test 추가.

Resolved updateRunRecord strict typed update

구현 계획은 run-record typed update가 unknown field를 거부해야 한다고 적는다. 이번 보완에서 registry input schema의 patch object를 .strict()로 바꾸고, unknown patch가 runtime-invalid-input으로 실패하는 테스트를 추가했다.

Remediation: tools/shared/runtime/operations.ts patch schema strict화, tests/runtime-operations.test.ts 회귀 테스트 추가.

Resolved renderReuseOutput 파일 쓰기 operation

계획은 renderReuseOutput(...)을 renderReuseTemplate 래핑과 output.md 쓰기, assertReuseTargetWriteAllowed 보호까지 포함한 operation으로 정의했다. 이번 보완에서 registry input에 projectRoot, runId, targetPath, explicitTargetPath, overwriteConfirmed를 추가하고, run output 및 명시 target 파일 쓰기를 연결했다.

Remediation: writeRenderedReuseOutput 추가, renderReuseOutput operation 확장, run output write test 추가.

Resolved extend-init.ts caller-provided timestamp

세션 리포트는 “타임스탬프/run-id는 operation 입력으로 받음”이라고 정리하면서도 후속 항목으로 extend-init.ts의 new Date()를 남겼다. 이번 보완에서 ensureExtendInitFiles가 generatedAt을 인자로 받도록 바꾸고, initPlugin의 입력값을 전달하게 했다.

Remediation: new Date() 제거, extend init tests가 정확한 generatedAt 값을 검증.

Documented P1/P2의 일부 계획 항목은 현재 spec과 구현 사이에서 정리 필요

.wonder/standards/*.md는 계획상 scaffold registry 대상이지만, 현재 govern init은 별도 하드코딩 starter text를 쓴다. 또한 updateConfig(domain, patch)는 계획에는 있지만 현재 authoritative §7 operation list에는 없다.

이 둘은 즉시 런타임 장애라기보다 문서 정합성 문제다. 계획을 구현 기준으로 계속 둘지, 현재 docs/deterministic-runtime.md의 §7 목록을 기준으로 둘지 정해야 한다.

Evidence: docs/deterministic-runtime-implementation-plan.md Phase 1 and Phase 2, tools/shared/runtime/govern-init.ts starter standards, docs/deterministic-runtime.md §7 operation list.

검증 명령 결과

  • 최종 검증은 보완 커밋 직전에 npm.cmd test, npm.cmd run typecheck, npm.cmd run validate, npm.cmd run drift, npm.cmd run check로 수행한다.
  • 초기 평가 당시에는 npm.cmd test 171 tests, 83 suites, 0 failures였다.

권장 조치

  1. updateConfig(domain, patch)를 실제 §7 operation으로 승격할지, 아니면 계획 문서에서 명시적으로 deferred로 둘지 결정한다.
  2. .wonder/standards/*.md starter content를 strict scaffold registry로 통합할지 결정한다.
  3. pre-commit hook에 npm test 또는 빠른 smoke subset을 추가할지 결정한다. 세션 리포트 자체가 테스트 미실행 회귀를 이미 기록하고 있다.

최종 결론

세션 리포트는 구현된 큰 구조를 잘 요약했고, 이번 보완으로 주요 계약 위반도 해결했다. 남은 항목은 updateConfig와 standards scaffold처럼 현재 §7 스펙에 없는 계획 잔여분을 제품 범위로 둘지 결정하는 문서/범위 정리다.
