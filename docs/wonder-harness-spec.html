이 문서의 성격 — 이 사양서는 현재 소스 코드를 역공학하여 도출한 정본이다. "무엇을 만들 계획인가"가 아니라 "플러그인이 지금 실제로 무엇인가"를 기술한다. v0.6.0은 실사용 중 발견된 세 가지 문제점(훅 미동작, templater 탐색 순서, handoff 스킬 경로)을 수정한 버전이다. AI용 압축 핸드오프 문서는 docs/wonder-harness-handoff.md 참조. ✦ 표시 섹션이 v0.6.0에서 변경되었다.

1. 개요

1.1 정체성

wonder-harness는 재사용 가능한 메커니즘 + 회사 표준 규칙이다. 특정 프로젝트의 복제본이 아니며, 고정된 엔터프라이즈 스택에서 도메인/화면 개발을 오케스트레이션하는 Claude Code 플러그인이다.

핵심 역할: 요청 문서(Request) → 4단계 에이전트 파이프라인(planner → templater → developer → ruler)으로 코드를 생성·수정하고, 규칙을 자동 검증한다.

📋 planner단계 1
→
🧩 templater단계 2
→
💻 developer단계 3
→
⚖️ ruler단계 4

1.2 책임 경계

영역소유자런타임 위치설명
규칙 (rules) 플러그인 ${CLAUDE_PLUGIN_ROOT}/rules/ 스택 정의·불변식·토큰 규약. ruler가 소유. 위젯 버전 종속 gotcha 박제 금지
메커니즘
(agents·hooks·commands·schema·requests)		 플러그인 ${CLAUDE_PLUGIN_ROOT}/ 에이전트 파이프라인·enforce 훅·카탈로그 스키마·커맨드·요청 양식
실 템플릿
(scaffold 파일·카탈로그)		 프로젝트 .claude/templates/ templater가 실사용 중 코드를 발견·토크나이즈하여 축적. 플러그인은 템플릿을 굽지 않음

2. 배포 · 의존 모델

2.1 마켓플레이스 구조

이 저장소는 단일 플러그인 마켓플레이스 구조다. 마켓플레이스 wrapper(0.1.0)와 플러그인(0.6.0)의 버전은 독립적으로 관리된다.

// .claude-plugin/marketplace.json (주요 필드)
{
  "name": "wonder-harness",
  "metadata": {
    "description": "Claude Code 개발 하네스 단일 플러그인 마켓플레이스",
    "pluginRoot": "./plugins"
  },
  "allowCrossMarketplaceDependenciesOn": ["claude-plugins-official"],
  "plugins": [
    {
      "name": "wonder-harness",
      "source": "./plugins/wonder-harness",
      "version": "0.6.0",        // ← 플러그인 버전 (plugin.json과 동기화 필수)
      "category": "Productivity"
    }
  ]
}
allowCrossMarketplaceDependenciesOn: "claude-plugins-official"를 명시하여 wonder-harness가 공식 마켓플레이스의 플러그인을 의존으로 선언할 수 있게 한다. 의존 플러그인 자동 활성화는 Claude Code v2.1.143 이상이 필요하다.

4. 파이프라인 계약

4.1 /wh-create — 신규 모듈 생성

alias: wonder-harness:wh-create
입력 기본 경로: .claude/requests/create_request.md

단계내용
① 입력 획득 인자 경로 우선, 기본 .claude/requests/create_request.md.
파일 없으면 플러그인 시드(${CLAUDE_PLUGIN_ROOT}/requests/create_request.md)를 복사하고 사용자에게 작성 안내 후 중단.
② 검증 게이트
(Fail-Fast)		 아래 4개 섹션이 모두 존재하고 비어있지 않아야 함(주석·공란 불가):
## 목표 · ## 범위 · ## 제약 · ## 수용 기준
미충족 시 누락 항목 보고 후 중단 (파이프라인 미시작).
③ 순차 디스패치 planner(create) → 모듈 분해·파일 경로·의존 순서
→ templater(create) → 카탈로그 우선 탐색, 신규 등록, index.json 갱신
→ developer(create) → 규칙 기반 코드 구현
→ ruler(review) → 5종 규칙 검증 보고
각 단계의 산출이 다음 단계의 입력이 됨.
④ 결과 단계별 요약 + ruler 검증 보고 (위반·개선 항목) 제시.

4.2 /wh-modify — 기존 모듈 수정

alias: wonder-harness:wh-modify
입력 기본 경로: .claude/requests/modify_request.md

구조는 /wh-create 와 동일하며, 모드가 modify이고 필수 섹션이 다르다:

검증 필수 섹션: ## 대상 · ## 변경 내용 · ## 영향 범위 · ## 수용 기준

modify 모드 파이프라인:

planner(modify)
→
templater(modify)
→
developer(modify)
→
ruler(review)

planner(modify) 는 기존 구조를 탐색하여 변경 영향 범위와 단계별 수정 계획을 산출한다.

4.3 /wh-review — 단일 에이전트 리뷰

alias: wonder-harness:wh-review
입력: 대상 경로 또는 설명 (예: src/.../UserService.java)

전체 파이프라인 없이 대상 유형에 따라 단일 에이전트를 review 모드로 호출한다:

대상 유형라우팅
.java / .kt / .html / .js 등 코드developer (review)
.claude/templates/**templater (review)
rules/**ruler (review)
모호한 경우사용자에게 1회 확인

5. 에이전트 사양

도구 권한 그래디언트: planner = Read · Grep · Glob · Write / templater · ruler = +Edit / developer = +Bash (유일)
에이전트단계입력산출도구
planner1 요청 문서 모듈 분해·파일 경로(7-file set)·의존 순서·위험 (YAGNI, 모호점 표시) Read Grep Glob Write
templater2 계획 카탈로그 우선 탐색 후 index.json 갱신(pathPatterns+경로), 토큰 규약 준수 템플릿 Read Grep Glob Write Edit
developer3 계획 + 템플릿 + 규칙 코드 생성/수정 (규칙 준수, SP-only·Kendo·ES6) Read Grep Glob Write Edit Bash
ruler4 developer 산출 규칙 5종 검증 보고 (위반·개선 항목) Read Grep Glob Write Edit

5.1 planner

요청 문서(create/modify)를 받아 도메인 단위 = 7-file set으로 분해한다:

// 5 Java 파일
{Entity}Controller.java
service/{Entity}Service.java
mapper/{Entity}Mapper.java
dto/{Entity}DTO.java
form/{Entity}Form.java

// 2 Frontend 파일
static/assets/js/front/wsmErp/{module}/{domainName}.js
templates/pages/wsmErp/{module}/{domainName}.html
모드동작
create목표/범위/제약 분석 → 모듈 분해, 의존성, 구현 순서, 위험 산출. 모호점 표시(추측 금지)
modify기존 구조 탐색(Read/Grep/Glob) → 변경 영향 범위, 단계별 수정 계획
review주어진 계획 검토 → 누락·모순·범위 적정성 지적 및 수정안

5.2 templater v0.6.0 변경

v0.5.x 문제: 카탈로그(index.json) 탐색을 건너뛰고 실 소스 코드를 직접 탐색하는 경향이 있었다. v0.6.0부터 카탈로그 탐색이 필수 1단계이며, 소스 탐색은 카탈로그 보완 용도로만 허용된다.

시작 시 필수 동작 (카탈로그 우선 탐색 프로토콜)

  1. [필수] 프로젝트의 .claude/templates/index.json 읽기. 없으면 플러그인 시드(${CLAUDE_PLUGIN_ROOT}/templates/index.seed.json)를 복사하여 부트스트랩.
  2. [필수] ${CLAUDE_PLUGIN_ROOT}/rules/templates.md 읽기 — 토큰 규약·치환 테이블·INDEX 포맷 로드.
  3. [필수] index.json 항목 전체를 검토하여 필요한 스캐폴드 매핑. 카탈로그에 적합한 항목이 있으면 해당 템플릿을 기반으로 구현.
  4. [보완] 카탈로그만으로 충분하지 않은 경우에 한해 실 프로젝트 소스 코드 탐색. 발견된 패턴은 토크나이즈하여 index.json에 신규 등록.
핵심 원칙: 소스 코드 탐색은 카탈로그의 보완이다. 카탈로그를 건너뛰고 소스 코드부터 탐색하는 것은 v0.6.0부터 위반으로 간주한다.
모드동작
create카탈로그 우선 탐색 → 없는 스캐폴드만 실 코드에서 토크나이즈 → index.json 신규 등록
modify카탈로그 기존 템플릿·패턴 수정, index.json 메타데이터(pathPatterns 포함) 갱신
review카탈로그 무결성 검증 — templates.md 체크리스트 기준
축적 철학: 플러그인은 템플릿을 사전 제작(bake)하지 않는다. 빈 시드에서 시작하여 실사용 중 발견된 코드 패턴을 토크나이즈해 카탈로그에 쌓는다. 현재 scaffolds/ 디렉터리는 비어있다(.gitkeep).

5.3 developer

시작 시 필수 동작 (규칙 로드):

${CLAUDE_PLUGIN_ROOT}/rules/security.md    // 항상

6. 훅 · 탐색강제 메커니즘 v0.6.0 변경

6.1 훅 등록

v0.5.x 문제: node \"경로\" 형태의 escaped 따옴표가 Windows cmd.exe 처리 방식과 충돌하여 훅이 실행되지 않았다. v0.6.0에서 따옴표를 제거하고 description·timeout 필드를 추가했다. 
// hooks/hooks.json (v0.6.0)
{
  "description": "wonder-harness template enforcement hooks",
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Read",
        "hooks": [{ "type": "command",
                    "command": "node ${CLAUDE_PLUGIN_ROOT}/hooks/scripts/mark-template-read.js",
                    "timeout": 5 }]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{ "type": "command",
                    "command": "node ${CLAUDE_PLUGIN_ROOT}/hooks/scripts/enforce-template.js",
                    "timeout": 10 }]
      }
    ]
  }
}
이벤트매처스크립트동작
PostToolUse Read mark-template-read.js index.json 읽기 감지 시 OS temp에 세션 마커 파일 생성 (비차단)
PreToolUse Write|Edit enforce-template.js 대상 경로가 카탈로그 pathPatterns에 매치 AND 마커 없으면 permissionDecision: "deny"
node 경로 전제: 훅 명령은 node가 PATH에 있어야 실행된다. Claude Code 자체가 Node.js 위에서 동작하므로 정상 설치 환경에서는 node가 PATH에 포함된다. 훅이 동작하지 않을 경우 node --version으로 PATH 확인 후, Node.js를 재설치하거나 PATH에 추가한다.

6.2 동작 흐름

세션 시작
   │
   ├── developer가 .claude/templates/index.json 읽음
   │       └── PostToolUse(Read) → mark-template-read.js
   │               → OS temp에 마커 파일 생성
   │                 (파일명: wh-template-checked-{safeSessionId})
   │
   └── developer가 코드 파일 Write/Edit 시도
           └── PreToolUse(Write|Edit) → enforce-template.js
                   → .claude/templates/index.json 로드
                   → decide(input, index, hasMarker(sessionId)) 호출
                          ├── 마커 있음 → null (허용)
                          ├── 경로 없음 → null (허용)
                          ├── 매치 없음 → null (허용)
                          └── 매치 있음 + 마커 없음 → deny (차단)

6.3 deny 결정 로직 (decide.js)

// lib/decide.js (핵심 로직)
function decide(input, index, markerPresent) {
  const filePath = input && input.tool_input && input.tool_input.file_path;

  if (!filePath) return null;                         // ① 경로 없음 → 허용
  if (!matchesTemplate(filePath, index)) return null; // ② 카탈로그 미매치 → 허용
  if (markerPresent) return null;                     // ③ 이미 탐색 완료 → 허용

  return {                                             // ④ 위 3개 아니면 → 차단
    hookSpecificOutput: {
      hookEventName: 'PreToolUse',
      permissionDecision: 'deny',
      permissionDecisionReason:
        '이 경로에 해당하는 템플릿이 카탈로그에 있습니다. ' +
        '파일 생성 전 .claude/templates/index.json 을 먼저 읽어 기존 템플릿을 탐색하세요.'
    }
  };
}

경로 매칭 (index-match.js + glob.js)

정규화(역슬래시→슬래시) 후 카탈로그의 모든 항목의 pathPatterns[]에 대해 glob 매칭:

// 지원하는 glob 구문 3가지
'**/'  →  (?:.*/)?   (0개 이상의 디렉터리)
'**'   →  .*         (임의 문자)
'*'    →  [^/]*      (슬래시 없는 단일 세그먼트)

// 앵커 전체 매치: new RegExp('^' + pattern + '$')

세션 마커 (marker.js)

// 마커 파일 경로: {os.tmpdir()}/wh-template-checked-{safeSessionId}
// safeSessionId: sessionId에서 [^a-zA-Z0-9_-] → '_' 치환

setMarker(sessionId)    // → 파일에 Date.now() 기록
hasMarker(sessionId)    // → fs.existsSync(...)
clearMarker(sessionId)  // → fs.unlinkSync(...) (정의됨, 현재 훅에서 미사용)

6.4 Fail-Open 설계

훅은 Fail-Open으로 설계되어 있다. 아래 어느 조건이라도 해당하면 항상 허용된다:
  • 카탈로그(.claude/templates/index.json) 파일이 없거나 파싱 오류
  • write 대상 파일 경로가 없음
  • 카탈로그의 어느 pathPatterns 에도 매치되지 않음 (빈 카탈로그 포함)
  • 해당 세션에서 이미 index.json을 읽음 (마커 있음)
  • 훅 스크립트 자체가 오류로 종료됨 (exit(0) → 조용히 허용)
deny 경로는 단 하나: (카탈로그 존재 + 경로 매치 + 세션 미탐색).

따라서 빈 시드({ "version":1, "templates":[] })로 시작하는 초기 상태에서는 enforce-template 훅이 아무것도 막지 않는다. 카탈로그에 항목이 축적될수록 탐색 강제가 점진적으로 발효된다.

11. 편의성·생산성 스킬 v0.6.0 변경

v0.6.0 재명명: 이전 버전의 "번들 스킬(Bundled Skills)"을 "편의성·생산성 스킬"로 재정의한다. 이 스킬들은 파이프라인(/wh-create, /wh-modify)과 독립적으로 동작하는 범용 생산성 도구다. harness 규칙 적용이 필요한 작업은 파이프라인 커맨드로 유도해야 한다 — 스킬이 대신하지 않는다.

11.1 handoff 스킬 저장 경로 규칙 v0.6.0 변경

v0.5.x 문제: handoff 스킬이 CLAUDE.md에 명시된 저장 경로를 무시하고 항상 OS 임시 디렉터리에 문서를 생성했다.

v0.6.0부터 handoff 스킬은 아래 우선순위로 저장 경로를 결정한다:

① 현재 프로젝트의 CLAUDE.md에 handoff 문서 저장 경로가 명시된 경우
   → 해당 경로에 저장

② CLAUDE.md에 경로 없음
   → OS 임시 디렉터리에 저장
      (파일명: wh-handoff-{timestamp}.md)
설계 원칙: 편의성·생산성 스킬은 프로젝트 컨텍스트(CLAUDE.md)를 존중해야 한다. 스킬 내부 지시가 프로젝트 설정보다 우선할 수 없다. 이 원칙은 모든 파일 생성·수정을 수반하는 스킬에 적용된다.
