wh-init 재설계 — 디자인 명세서

📅 2026-06-05 ✅ 구현 승인됨 📦 wonder-harness 플러그인

1. 문제 정의

현재 /wh-init은 메타룰 탐색 가이드를 활용하여 소스 코드를 탐색하고, 프로젝트별 규칙을 생성합니다. 동작은 하지만 두 가지 핵심 문제가 있습니다:

  1. "왜(Why)" 컨텍스트 부재. ruler는 어떤 컨벤션이 존재하는지는 파악하지만, 그 뒤에 있는 아키텍처 결정의 이유는 알지 못합니다. ADR 컨텍스트 없이 생성된 규칙은 기존 설계 제약과 충돌하거나, 우연히 생긴 패턴을 규칙으로 굳힐 수 있습니다.
  2. 강제 실행 메커니즘 부재. init은 권고사항일 뿐입니다. 개발자가 초기화하지 않은 프로젝트에서 /wh-create를 실행하는 것을 막을 방법이 없어, 규칙을 인식하지 못하는 결과물이 생성됩니다.

2. 목표

  • 관찰 가능한 컨벤션과 추론된 아키텍처 의도를 모두 반영하도록, ADR 역공학을 규칙 생성의 필수 선행 단계로 추가
  • 훅(Hook) 강제를 통해 init 완료를 /wh-create 및 /wh-modify의 하드 게이트로 설정
  • /wh-init 내부에서도 단계 순서를 훅 수준에서 강제 (커맨드 수준이 아닌)
  • 레이어별로 타임스탬프 버전이 지정된 사람이 읽을 수 있는 HTML 보고서 생성

3. 결정 사항

주제결정
훅 강제 범위C — 내부 단계 순서 강제 + 교차 명령 게이트
ADR 저장C — .claude/adr/에 영구 저장, 향후 규칙 업데이트에 재사용
HTML 보고서 버전 관리B+C — 레이어별 타임스탬프 버전 관리
상태 추적 방식B — .claude/.wh-state.json을 단일 진실 원천으로 사용

4. 새로운 Init 흐름

/wh-init [--backend] [--frontend] [--security] [--templates]
         (플래그 없음 = 4개 레이어 전체)
0

요청 시드 복사 기존

init-requests.js가 create_request.md + modify_request.md를 .claude/requests/로 복사 (이미 존재하면 건너뜀). 레이어 루프 이전에 한 번만 실행됩니다.
→ 상태 기록: requests_copied = true

▼ 선택된 각 레이어에 대해 순차적으로 반복:

1

ADR 역공학 신규

ruler (adr-extract 모드)가 프로젝트 소스를 탐색합니다.
→ .claude/adr/{layer}.md 작성
→ 상태 기록: adr.{layer} = ISO-timestamp

2

규칙 생성 개선

ruler (generate 모드)가 메타룰 + .claude/adr/{layer}.md를 읽습니다.
→ .claude/rules/{layer}.md 작성
→ 상태 기록: rules.{layer} = ISO-timestamp

3

HTML 보고서 신규

ruler가 HTML 요약을 생성합니다.
→ .claude/reports/wh-init-{layer}-YYYYMMDD-HHMMSS.html 작성
→ 상태 기록: reports.{layer} = filename

단계는 반드시 순서대로 실행되어야 합니다. 훅이 해당 레이어의 Step 1이 상태에 기록되지 않으면 Step 2를 차단하고, Step 2가 기록되지 않으면 Step 3을 차단합니다.

5. 상태 파일 스키마

경로: .claude/.wh-state.json

{
  "version": 1,
  "requests_copied": true,
  "adr": {
    "backend":   "2026-06-05T10:00:00Z",
    "frontend":  null,
    "security":  null,
    "templates": null
  },
  "rules": {
    "backend":   "2026-06-05T10:05:00Z",
    "frontend":  null,
    "security":  null,
    "templates": null
  },
  "reports": {
    "backend":   "wh-init-backend-20260605-100500.html",
    "frontend":  null,
    "security":  null,
    "templates": null
  }
}

불변 조건

  • rules.{layer}는 adr.{layer}가 설정되기 전에 설정될 수 없습니다.
  • reports.{layer}는 rules.{layer}가 설정되기 전에 설정될 수 없습니다.
  • 상태 파일이 훅의 단일 진실 원천입니다. 아티팩트 존재 여부(.claude/adr/*.md, .claude/rules/*.md)는 상태 파일이 없거나 손상되었을 때 보조 가드로 확인됩니다.

6. 훅 강제 설계

6a. 교차 명령 게이트 — init 전에 wh-create / wh-modify 차단

훅: PreToolUse on Write|Edit
스크립트: hooks/scripts/enforce-init.js

if file_path가 .claude/ 하위도 아니고 플러그인 루트 하위도 아니면:
  .claude/.wh-state.json 읽기
  if 상태 파일이 완전히 없으면:
    거부: "wonder-harness가 초기화되지 않았습니다. /wh-init을 먼저 실행하세요."
  if state.rules에 null이 아닌 레이어가 하나도 없으면:
    거부: "완전히 초기화된 레이어가 없습니다. /wh-init [--layer]를 먼저 실행하세요."
  → 허용 (부분 초기화는 허용됨)
부분 초기화 정책: --backend만 실행된 경우에도 게이트는 통과됩니다. ruler의 validate 단계(wh-create / wh-modify 파이프라인 끝)가 이미 레이어별로 누락된 규칙 파일을 플래그하고 wh-init --{layer} 실행을 권장합니다. 교차 명령 게이트의 역할은 완전히 초기화되지 않은 사용을 방지하는 것이지, 4개 레이어 전체 완성을 강제하는 것이 아닙니다.

6b. Init 내부 단계 순서 강제 — 순서 외 쓰기 차단

동일 훅, enforce-init.js, 추가 로직:

if file_path가 .claude/rules/{layer}.md와 일치하면:
  if state.adr.{layer}가 null이면:
    거부: "{layer}의 ADR이 먼저 완료되어야 합니다.
           Step 1 (adr-extract)을 먼저 완료하세요."

if file_path가 .claude/reports/wh-init-{layer}-*.html과 일치하면:
  if state.rules.{layer}가 null이면:
    거부: "{layer}의 규칙이 먼저 생성되어야 합니다.
           Step 2 (generate)를 먼저 완료하세요."

6c. 상태 기록 유틸리티

스크립트: hooks/scripts/write-state.js

init-requests.js가 사용하며 각 ruler 단계가 완료된 후 커맨드가 호출하는 공유 모듈입니다. 상태 파일의 원자적 읽기-수정-쓰기를 처리합니다.

7. ruler 에이전트 변경 사항

새로운 서브 모드: adr-extract

각 레이어의 Step 1로 wh-init이 호출합니다.

입력:

  • 레이어 이름 (backend / frontend / security / templates)
  • 프로젝트 소스 파일 (Glob + Grep 탐색 활용)

처리 과정:

  1. generate 모드와 동일한 탐색 가이드를 사용하여 해당 레이어의 프로젝트 코드를 탐색
  2. 발견된 각 주요 패턴에 대해 다음을 추론:
    • 컨텍스트 — 이 패턴이 해결하는 문제
    • 결정 — 어떤 아키텍처 선택이 이루어졌는지
    • 근거 — 왜 이 선택인지 (코드 구조, 네이밍, 주석에서 추론)
    • 결과 — 이 결정이 향후 코드에 부과하는 제약
  3. 추론 결과를 3~7개의 ADR 항목으로 그룹화. 노이즈(예: 파일 하나에만 나타나는 패턴)는 제거
  4. 사용자에게 요약을 제시하여 수정 확인
  5. .claude/adr/{layer}.md 작성

ADR 문서 형식:

# ADR: {Layer} — {프로젝트 이름}
생성일: {ISO 날짜}

## ADR-{N}: {짧은 제목}
**컨텍스트:** ...
**결정:** ...
**근거:** ...
**결과:** ...

개선된 generate 모드

Step 2에서 이제 규칙 초안 작성 전에 .claude/adr/{layer}.md를 읽습니다.

현재 대비 추가 입력:

  • .claude/adr/{layer}.md — 이 레이어의 ADR (필수; 없으면 중단)

프로세스 변경:

  • 메타룰 로드(Step 1) 후 ADR을 로드
  • 초안 작성(Step 3) 중 각 규칙 제약을 ADR 결과와 교차 참조. 충돌 플래그
  • 사용자 확인(Step 4)에서 ADR-규칙 충돌을 표면화하여 해결

8. HTML 보고서 구조

경로: .claude/reports/wh-init-{layer}-YYYYMMDD-HHMMSS.html

섹션 (순서대로):

  1. 헤더 — 레이어 이름, 프로젝트 이름, 생성 타임스탬프
  2. ADR 요약 — 추출된 모든 ADR의 컨텍스트/결정/근거 컬럼 테이블
  3. 규칙 스냅샷 — 생성된 .claude/rules/{layer}.md를 포맷된 HTML로 렌더링
  4. ADR ↔ 규칙 매핑 — 어떤 ADR 항목이 어떤 규칙 섹션에 영향을 미쳤는지
  5. 충돌 사항 — 메타룰 기본값과 충돌한 ADR 결과 및 해결 방법
  6. 푸터 — wonder-harness 버전, 플러그인 루트 경로
보고서는 자체 포함 HTML(인라인 CSS, 외부 리소스 없음)로, 파일 시스템에서 직접 열 수 있습니다.

10. 범위 외

  • ADR 수동 작성 (사용자가 .claude/adr/*.md를 직접 편집)
  • 필수 레이어 구성 (.claude/.wh-config.json) — 향후 작업
  • 규칙 재실행 없이 ADR 단계만 다시 실행
  • HTML 보고서 아카이빙을 위한 CI/CD 통합
