훅 기반 파이프라인 강제

wonder-harness 설계 보고서  |  작성일: 2026-06-06  |  대상 버전: 0.5.x  |  대상 커맨드: wh-create · wh-modify · wh-init

문제: LLM 지시는 보장이 아니다

wonder-harness의 파이프라인은 네 에이전트가 순서대로 실행되는 체인 구조입니다.

planner → templater → developer → ruler

이 순서가 커맨드 파일의 마크다운 지시문으로만 표현될 경우, LLM은 지시를 따르려 하지만 아무것도 강제하지 않습니다. 실제로 두 가지 실패 패턴이 관찰됩니다.

실패 패턴발생 상황결과
단계 건너뜀 templater가 완료되기 전에 developer가 코드를 작성 템플릿이 활용되지 않음. 일관성 검증 무의미
커맨드 순서 무시 wh-init이 완료되기 전에 wh-create 실행 프로젝트 규칙 없이 코드 생성. ruler가 메타 규칙만으로 검증
핵심 문제 두 실패 모두 조용히 발생합니다. 파이프라인은 완료된 것처럼 보이지만, 품질 게이트는 이미 우회됐습니다.

해결책: 훅을 물리적 게이트로

Claude Code 훅은 OS 수준에서 툴 호출을 가로챕니다. PreToolUse 훅이 permissionDecision: "deny"를 반환하면, LLM의 지시와 무관하게 파일 쓰기가 물리적으로 불가능해집니다.

⚠ Before — 지시 기반 (권고)

"Claude, 이 순서를 따르세요."
→ LLM이 무시하거나 잊으면 뚫림

✓ After — 훅 기반 (강제)

"선행 상태가 없으면 파일 시스템이 거부한다."
→ 상태가 없으면 물리적으로 쓸 수 없음

핵심 통찰 의미 있는 파이프라인 동작은 결국 파일 쓰기로 귀결됩니다. templater는 index.json을 씁니다. developer는 Java/HTML/JS 파일을 씁니다. ruler는 검증 보고서를 씁니다. 각 쓰기는 가로챌 수 있는 체크포인트입니다.

아키텍처

3.1 상태 기계 — .wh-state.json

모든 파이프라인 상태는 프로젝트 루트의 .claude/.wh-state.json에 저장됩니다. 훅 스크립트와 에이전트 모두 이 파일을 통해 통신합니다. 

{
  "version": 1,
  "current": {
    "command": "wh-create",   // 현재 활성 커맨드
    "domain":  "eqEquipment"  // 현재 작업 도메인 (planner가 기록)
  },
  "wh-create": {
    "planner":   "done",  // 완료 → done / 미완료 → null
    "templater": "done",
    "developer": null,
    "ruler":     null
  },
  "wh-modify": { ... },
  "adr":     { "backend": "2026-06-06T10:00:00Z", ... },
  "rules":   { "backend": "2026-06-06T10:05:00Z", ... },
  "reports": { "backend": "wh-init-backend-20260606-100500.html", ... }
}

상태 전환은 에이전트가 write-state.js를 통해 직접 기록합니다. 에이전트 자신이 자신의 완료를 가장 잘 알기 때문입니다. 

node "${CLAUDE_PLUGIN_ROOT}/hooks/scripts/write-state.js" "<cwd>" "wh-create.planner" "done"

3.2 훅 스크립트 구성

세 개의 enforce 스크립트가 PreToolUse(Write|Edit)에 순서대로 등록됩니다.

PreToolUse(Write|Edit) 모든 파일 쓰기/수정 시도 시 발화
enforce-init.js wh-init 단계 순서 강제 + 전체 커맨드 cross-gate (rules 파일 존재 확인)
enforce-create.js wh-create 파이프라인 단계 강제 + 템플릿 탐색 게이트 (current.command=wh-create일 때만 활성)
enforce-modify.js wh-modify 파이프라인 단계 강제 + 템플릿 탐색 게이트 (current.command=wh-modify일 때만 활성)

각 스크립트는 state.current.command가 자신의 커맨드와 다르면 즉시 allow로 종료합니다. 커맨드 간 간섭이 없습니다.

3.3 스크립트별 강제 규칙

enforce-init.js — active 커맨드와 무관하게 항상 실행

쓰려는 파일선행 조건
.claude/rules/{layer}.mdstate.adr.{layer} non-null
.claude/reports/wh-init-{layer}-*.htmlstate.rules.{layer} non-null
비.claude/ 코드 파일 (전체)임의 레이어의 state.rules.{layer} non-null

enforce-create.js — current.command === "wh-create"일 때만 활성

쓰려는 파일선행 조건
.claude/templates/**wh-create.planner === "done"
코드 파일 (비.claude/)wh-create.templater === "done"
코드 파일 + 템플릿 카탈로그 매칭위 조건 + 세션 내 index.json 읽기 완료
.claude/reports/wh-create-*wh-create.developer === "done"

enforce-modify.js — current.command === "wh-modify"일 때만 활성. enforce-create.js와 동일 구조, 보고서 경로만 wh-modify-*로 상이.

파이프라인 전체 흐름

Step 0 → planner planner=done ✓ → templater templater=done ✓ → developer developer=done ✓ → ruler

각 단계 사이의 게이트(초록 태그)는 enforce-create.js가 PreToolUse 훅으로 강제합니다.

단계 건너뜀 시 실제 동작

templater 완료 전에 developer가 코드 파일 쓰기를 시도하면: 

PreToolUse(Write) 발화
  대상 파일: src/main/java/io/boot/wonder/web/eq/EqEquipmentController.java

enforce-create.js 체크:
  state.current.command === "wh-create"       → ✓ 활성
  isClaudeInternal(filePath) === false         → 코드 파일 게이트 진입
  state["wh-create"].templater === null        → 선행 조건 미충족

응답:
  permissionDecision: "deny"
  permissionDecisionReason: "[wh-create] Templater has not completed.
    Run the templater agent before writing code files."

→ 파일 쓰기 거부. LLM은 진행 불가.

커맨드 초기화 — 항상 처음부터

커맨드는 실행 시작 시 파이프라인 상태를 즉시 초기화합니다. 완료 시점이 아닌 시작 시점에 초기화하는 이유:

  • 중단된 이전 실행의 부분 상태가 다음 실행의 게이트를 silently 열어주는 현상을 방지
  • 요청 파일이 변경된 후 재실행해도 planner부터 다시 시작함을 보장
  • 상태 추론 없이 예측 가능한 동작 확보

wh-create.md Step 0 초기화 시퀀스:

node write-state.js "<cwd>" "current.command"    "wh-create"
node write-state.js "<cwd>" "wh-create.planner"   null
node write-state.js "<cwd>" "wh-create.templater" null
node write-state.js "<cwd>" "wh-create.developer" null
node write-state.js "<cwd>" "wh-create.ruler"     null

에이전트 책임 — 상태 신호

각 에이전트는 자신의 작업 완료 후 write-state.js를 직접 호출해 상태를 기록합니다. 에이전트 지시문의 Pipeline State Signal 섹션에 명시됩니다.

에이전트읽는 상태기록하는 상태
planner — current.domain, {command}.planner=done
templater current.command {command}.templater=done
developer current.command {command}.developer=done
ruler current.command, current.domain 보고서 파일 출력, {command}.ruler=done
설계 원칙 완료를 가장 잘 아는 주체(에이전트 자신)가 완료를 선언합니다. 커맨드나 외부 훅이 에이전트 내부 완료를 추론하지 않습니다.

템플릿 탐색 게이트 흡수

기존 독립 스크립트였던 enforce-template.js의 로직은 enforce-create.js와 enforce-modify.js에 흡수됐습니다.

흡수 후 게이트 우선순위:

developer가 코드 파일 쓰기 시도
  → templater=done? NO  → DENY: "Run templater first"  ← 단계 게이트 우선
  → templater=done? YES
      → 템플릿 카탈로그 매칭? YES + 세션 마커 없음?
          → DENY: "Read index.json first"               ← 탐색 게이트
      → 매칭 없음
          → ALLOW

독립 훅 등록이 사라지고, 에러 메시지에 "어느 단계가 선행되어야 하는가"가 포함됩니다.

강제 범위 경계

강제하는 것강제하지 않는 것
wh-create / wh-modify 내 단계 순서각 단계 산출물의 품질·정확성
코드 쓰기 전 템플릿 탐색올바른 템플릿이 선택됐는지
코드 쓰기 전 wh-init 선행 여부생성된 규칙이 코드에 올바르게 적용됐는지
ruler 보고서 쓰기 전 developer 완료ruler 검증 결과의 정확성
wh-modify 훅의 wh-create 간섭 방지다중 도메인 병렬 작업
명확한 한계 훅은 순서와 존재를 강제합니다. 정확성은 ruler 에이전트의 책임입니다.

설계 결정 요약

결정 항목선택근거
강제 메커니즘 PreToolUse 훅 deny 물리적 게이트. LLM 지시로 우회 불가
상태 저장 .wh-state.json + 명시적 플래그 wh-init의 검증된 패턴 재사용. 세션 재시작 후에도 유지
상태 기록 주체 각 에이전트가 write-state.js 직접 호출 완료를 가장 잘 아는 주체가 선언
커맨드 재실행 동작 항상 처음부터 (always-from-scratch) 부분 상태가 silently 게이트를 여는 현상 방지
커맨드 간 격리 current.command 체크 후 즉시 종료 각 enforce 스크립트가 자신의 커맨드에만 반응
훅 통합 방식 커맨드별 3개 스크립트 (enforce-init, enforce-create, enforce-modify) 커맨드당 하나의 스크립트. 소유권 명확. 로직 누수 없음
템플릿 게이트 enforce-create/modify에 흡수 별도 훅 등록 제거. 단계 인식 에러 메시지 가능
