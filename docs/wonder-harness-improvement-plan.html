wonder-harness 개선 계획

기존 WONDER 프로젝트 분석 → 플러그인 적합화 (grill 인터뷰 합의본)
Spring Boot · MyBatis(SP) · Thymeleaf · Kendo 버전 0.2.0 → 0.3.0 2026-05-29

🎯 목적

기존 WONDER 프로젝트(raw/)의 검증된 규칙을 플러그인에 원칙 수준으로 반영해, 플러그인을 그 스택에 바로 투입할 수 있게 적합시킨다. 단, 플러그인다움(재사용 가능한 하네스)을 잃지 않는다.

플러그인을 개발하며 기존 프로젝트를 분석하는 이유 — 프로젝트의 규칙을 플러그인에 반영해 즉시 사용 가능하게 만들기 위함.

핵심 경계 — 무엇이 어디에 사는가

구분소유 / 위치내용
규칙 플러그인
${CLAUDE_PLUGIN_ROOT}/rules/		 스택 정의·불변식·토큰 규약 = 회사표준. 에이전트가 여기서 로드. 위젯 버전 종속 gotcha는 넣지 않음
실 템플릿 프로젝트
.claude/templates/ + index.json		 templater가 실사용 중 발견·생성해 축적. 플러그인은 템플릿 파일을 굽지 않음
메커니즘 플러그인 에이전트 파이프라인 · enforce 훅 · 카탈로그 스키마 · 커맨드 · 요청양식

＊ "정본"의 의미 = 시드 원천 소유권이지 런타임 독점이 아니다. 프로젝트는 자기 관례의 주인. raw/.claude/는 분석 입력일 뿐 수정하지 않는다.

왜 이렇게 — 발견된 불일치

현재 플러그인은 추측한 스택으로 만들어져 실제 프로젝트와 어긋나 있었다.

항목플러그인 (기존 가정)실제 프로젝트
영속성Repository, "JPA 또는 MyBatis"MyBatis + SQL Server 저장프로시저 전용
3계층 이름RepositoryMapper + 별도 DTO·Form
패키지com.wonderitio.boot.wonder
프론트JSP / jQueryThymeleaf + Kendo 웹컴포넌트 + ES6

템플릿비어있음 (.gitkeep)레이어별 문서화된 풍부한 세트

플러그인에 적용할 교정

규칙 5종 rules/

파일핵심 교정
backend.md Repository→Mapper, JPA→SP-only(EXEC dbo.SP_…, CALLABLE), DTO(BaseDTO·@GridSetup)·Form(createdRows/updatedRows/deletedRows) 분리, delete→insert→update, errorCode "E"→예외, 패키지 io.boot.wonder
frontend.md jQuery→Thymeleaf+Kendo+ES6, 그리드 ID 규약, searchForm 4-hidden, changesData 직렬화 원칙(deletedRows 필터링), 저장 순서
security.md 재고 무결성 SP-only, hasMenuRoleType 권한 매핑표, FileUtils 확장자 검증, SecurityUtils 컨텍스트
workflow.md 신설 도메인 네이밍, 구현 전 템플릿 탐색 필수(훅의 규범 근거), DB 레이블 등록
templates.md 교체 template-meta 교체 — 토큰 규약·치환표·INDEX 포맷·검증 체크리스트

＊ 상호참조 정리: 규칙↔규칙 = ${CLAUDE_PLUGIN_ROOT}/rules/… / 규칙→템플릿 = 프로젝트 로컬 .claude/…

에이전트 4종 agents/

developer
  • 스택 교정: SP-only·Kendo·ES6
  • 규칙 5종을 플러그인 루트에서 로드
  • 템플릿을 프로젝트 index.json에서
templater
  • 단일 index.json 유지
  • raw 토큰 규약 준수
  • 실코드에서 템플릿 축적
ruler
  • 규칙 5종 소유(workflow 추가)
  • 검증 체크리스트 갱신
planner
  • 도메인 단위 = 7파일 세트 인지
  • 경미한 조정

메타데이터

  • plugin.json · marketplace.json — 스택 반영 + 버전 0.2.0 → 0.3.0 동기화
  • CLAUDE.md · README.md — 스택·구조 설명 갱신
  • 커밋 분리: feat/fix: 변경 → 별도 chore: bump version to 0.3.0
  • 훅 JS · 빈 시드 — 변경 최소(카탈로그가 비어 시작, 패턴 축적 시 작동)

하지 않을 것

  • raw 템플릿 파일을 플러그인에 굽기
  • 프로젝트에 플러그인 전량 시딩 / wh-init 신설
  • raw/.claude/ 수정
  • 위젯 버전 종속 gotcha를 규칙에 박제
  • 훅 JS 대규모 개조

검증

npm run validate→ 기존 훅 테스트 회귀→ WONDER 사본에 /wh-create 실행

  • ruler가 교정된 5종 규칙으로 검증하는가
  • developer가 SP-only/Kendo/올바른 패키지로 생성하는가

＊ "테스트하며 템플릿 축적"이 곧 통합검증이 된다.
